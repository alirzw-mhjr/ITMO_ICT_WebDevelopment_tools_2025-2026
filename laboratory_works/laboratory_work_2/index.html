    Лабораторная работа 2. Потоки. Процессы. Асинхронность

    Цель работы

    Понять отличия между потоками (threading) и процессами (multiprocessing) и разобраться, что такое асинхронность (asyncio) в Python: когда каждый из подходов даёт ускорение, а когда нет.

    Ключевая теория

    В CPython есть GIL (Global Interpreter Lock) — глобальная блокировка, из-за которой в один момент времени байт-код Python исполняет только один поток. Отсюда вытекает главный принцип выбора подхода:

    Подход Параллелизм Где выигрывает
    threading Потоки в одном процессе, общий GIL I/O-bound задачи
    multiprocessing Отдельные процессы, у каждого свой GIL CPU-bound задачи
    asyncio Кооперативная многозадачность, 1 поток I/O-bound задачи
    • CPU-bound задача упирается в вычисления процессора. Реальное ускорение даёт только multiprocessing, потому что разные процессы выполняются на разных ядрах и не делят один GIL.
    • I/O-bound задача почти всё время ждёт (сеть, диск). Здесь помогают threading и asyncio: пока одна задача ждёт ответ, выполняется другая.

    Задача 1. Подсчёт суммы чисел (CPU-bound)

    Три программы считают сумму всех чисел от 1 до N. Диапазон делится на несколько равных частей, каждая часть считается своей подзадачей. Функция calculate_sum() во всех вариантах выполняет один и тот же расчёт суммы на отрезке.

    Файлы: task1/threading_sum.py, task1/multiprocessing_sum.py, task1/async_sum.py.

    threading

    Каждый отрезок считается в отдельном threading.Thread, результат пишется в общий список. Из-за GIL потоки на чистых вычислениях выполняются по очереди, поэтому ускорения почти нет.

    def calculate_sum(start, end, results, index):
    total = 0
    for number in range(start, end + 1):
        total += number
    results[index] = total

    multiprocessing

    Каждый отрезок считается в отдельном процессе через multiprocessing.Pool. У каждого процесса свой интерпретатор и свой GIL, поэтому вычисления реально идут параллельно на разных ядрах — это даёт настоящее ускорение.

    with multiprocessing.Pool(processes=workers) as pool:
    partials = pool.map(calculate_sum, ranges)
total = sum(partials)

    async

    Каждый отрезок оформлен как корутина, запуск через asyncio.gather. Но asyncio — это один поток и кооперативная многозадачность: без точек await внутри цикла корутины не «уступают» управление, и вычисления идут фактически последовательно. Для CPU-bound задачи асинхронность смысла не имеет.

    Результаты замеров

    Замеры выполнены при N = 10^8 и 4 подзадачах (полное N = 10^13 в чистом Python считается слишком долго; структура программ при этом не меняется).

    Подход Время выполнения Ускорение vs threading
    threading 3.090 с ×1.0
    multiprocessing 1.264 с ×2.4
    asyncio 3.117 с ×1.0

    Вывод по задаче 1. На CPU-bound задаче побеждает multiprocessing — он единственный обходит GIL и задействует несколько ядер. threading и asyncio показывают практически одинаковое время на уровне последовательного исполнения, потому что вычисления не отпускают GIL и не содержат точек ожидания.

    Задача 2. Параллельный парсинг веб-страниц (I/O-bound)

    Три программы параллельно загружают страницы книг с Project Gutenberg, парсят заголовок и автора и сохраняют их в таблицу books базы данных из лабораторной работы 1.

    Функция parse_and_save(url) во всех вариантах: загружает HTML по URL, парсит заголовок, сохраняет его в БД и печатает результат.

    Файлы: task2/threading_parser.py, task2/multiprocessing_parser.py, task2/async_parser.py. Общие модули: task2/db.py (подключение к laboratory_work_1/app.db и save_book()), task2/parser_utils.py (загрузка списка URL и parse_title()).

    Связь с базой данных из ЛР 1

    Парсеры пишут в ту же таблицу books, что используется в сервисе буккроссинга: из страницы книги извлекаются заголовок (title) и автор (authors). Перед вставкой проверяется, нет ли уже книги с таким заголовком, чтобы повторные запуски не создавали дублей.

    def save_book(title, authors=None):
    with SessionLocal() as session:
        if session.scalar(select(Book).where(Book.title == title)):
            return False
        session.add(Book(title=title, authors=authors))
        session.commit()
        return True

    threading

    Список URL делится на равные части по числу потоков, каждый поток последовательно обрабатывает свою часть. На сетевых запросах GIL отпускается во время ожидания ответа, поэтому потоки работают одновременно и дают ускорение.

    multiprocessing

    Список URL делится на равные части, каждую часть обрабатывает отдельный процесс. Запросы идут параллельно, но запуск процессов и их обслуживание дороже, чем у потоков. Каждый процесс открывает свою сессию к SQLite, а timeout в db.py защищает от ошибки database is locked при одновременной записи.

    async

    Все запросы — корутины, запускаются через asyncio.gather в одном потоке с библиотекой aiohttp. Пока один запрос ждёт сеть, выполняются другие, поэтому десятки страниц грузятся почти одновременно без накладных расходов на потоки и процессы. Число одновременных запросов ограничено семафором (CONCURRENCY = 10), чтобы сервер не разрывал часть соединений.

    async def parse_and_save(session, semaphore, url):
    async with semaphore:
        async with session.get(url, timeout=aiohttp.ClientTimeout(total=30)) as resp:
            html = await resp.text()
    title, author = parse_title(html)
    save_book(title, author)

    Результаты замеров

    Замеры выполнены на 30 URL. threading/multiprocessing — 4 воркера, async — семафор на 10 одновременных запросов.

    Подход Время выполнения Ускорение vs threading
    threading 64.85 с ×1.0
    multiprocessing 56.97 с ×1.1
    asyncio 19.69 с ×3.3

    Время зависит от скорости ответа gutenberg.org и может меняться от запуска к запуску, но соотношение между подходами сохраняется.

    Вывод по задаче 2. На I/O-bound задаче побеждает asyncio: один поток обслуживает много одновременных запросов с минимальными накладными расходами и легко поднимает уровень параллелизма (10 против 4). threading тоже ускоряет работу относительно последовательного парсинга, но ограничен числом потоков и расходами на переключение. multiprocessing справляется с задачей, однако для ожидания сети он избыточен: запуск процессов и межпроцессное взаимодействие делают его самым тяжёлым из трёх вариантов и почти не быстрее threading.

    Общий вывод

    • CPU-bound (задача 1) → multiprocessing: только отдельные процессы обходят GIL и используют несколько ядер.
    • I/O-bound (задача 2) → asyncio (или threading): пока задача ждёт ввод-вывод, процессор занимается другими задачами; asyncio делает это дешевле всего.
    • Универсального «самого быстрого» подхода нет — выбор зависит от природы задачи. GIL делает threading бесполезным для вычислений, но эффективным для ожидания, а asyncio доводит идею кооперативного ожидания до максимума.
